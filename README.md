## 概要
 
`coupon-api` は、クーポンシステム刷新（NCP-2892）における **実行時（ランタイム）側のAPI面**を担うサービスです。クーポンと配布ルールを保持し、運用時に他システムから叩かれる複数のAPIを提供します。
 
クーポンの管理（登録・編集・承認・テンプレート・CSV 等）を担う管理コンソール側のバックエンドは、本リポジトリではなく [`coupon-admin`](#関連リポジトリ) が担当します。両者は障害の影響を相互に及ばさないため分離し、`coupon-api` から `coupon-admin` のAPIを呼ぶことはしません。
 
## 設計方針
 
- **会員×クーポンの事前紐づけを持たない。** 従来の「配信時に会員IDとクーポンIDを紐づけてレコードを作る」方式は、会員数・クーポン数の増加に比例して処理時間が伸びるため廃止。配布ルールを保持し、表示時に該当判定する方式に変更する。
- **会員属性は本サービスに持たせない。** 年齢・性別・行動フラグ等の会員属性は、クーポン専用ではない独立した**ユーザプロファイルストア（別プロダクト）**が保持する。本サービスは会員ドメインと密結合しない。
- **セグメント該当判定は逆引きで行う。** 「属性を投げると、その属性に当たるクーポン集合が返る」形。配布ルールをループして都度 SF/DMP へ問い合わせて照合する方式は取らない（OpenSearch percolator を想定）。
- **会員文脈の取りまとめは BFF の責務。** プロファイルストアからの属性取得、消費済み・選択状態の差し引き、並び順・画像（CMS）の整形は BFF が行う。

## アーキテクチャ（モジュラーモノリス）

本サービスは **単一デプロイのモジュラーモノリス**。中を2つの境界づけられたコンテキストに分ける — **eligibility**（会員に依存しない該当判定エンジン）と **member-coupon-state**（選択・お気に入り・利用済み等、会員IDをキーにする状態）。

会員キーの状態を別サービスに切り出さないのは意図的。分散は最後の手段であり、物理分割の必要（障害隔離・独立スケール・チーム/デプロイ独立）が**実証されるまではコストだけが先行**するため。境界はパッケージ・データ所有・依存方向で守り、分割が要る段になったらモジュールごと引き剥がす。

不変条件（詳細・抽出トリガーは [`docs/DESIGN.md`](docs/DESIGN.md) の「アーキテクチャ方針: モジュラーモノリス」）:

- **別モジュール・別データ所有** — 各モジュールは自分のストアだけを読み書きする。
- **依存は一方向**（member → eligibility のみ。逆向き import は禁止し、ArchUnit がビルド毎に強制）。
- **差し引きは合成層でのみ** — 消費済み・停止中の除外は合成層が行い、純粋な eligibility は生集合を返す。

## 主なAPI（面）
 
刷新後に本リポジトリが提供する想定のAPI群（詳細・確定は NCP-2892 のAPI仕様を正とする）。
 
- クーポン一覧取得（属性に基づく該当判定＋全員配信の付与）
- クーポン選択更新 / クーポンお気に入り更新
- 使用済み・顧客選択クーポン取得 / クーポン利用更新（POS）
- ウェルカムクーポン登録 / イベント型クーポン登録
- クーポン詳細情報取得 / 全件取得、親カテゴリ情報取得、並び順一覧取得
- 店舗用クーポン詳細情報取得（PIT）、商品紐づきクーポンID一覧取得（在庫）
- クーポン存在確認（CMS）、クーポン登録可否（Smooth）、クーポン情報取得（SMC）
- 顧客別クーポン移行 / 統合 / 削除（ID統合）
- セグメント配布API（SMC / DMP・新設）
> 注: `member-agnostic` は「会員×クーポンの事前紐づけ・会員属性を保持しない」という意味であり、消費済み・選択状態など会員IDをキーに扱うAPIは本サービスに含まれる。属性の保有と紐づけ生成をしない、が原則。
 
## 該当判定の考え方（percolator）
 
配布ルール（例: 20〜30代）をあらかじめ「条件」として保存しておき、1人分の属性を投げると当たる条件（＝クーポン）が返る、という通常検索と逆向きの仕組みを用いる。これにより、ルールのループ照合が不要になり、配信件数も集計機能で取得できる。範囲条件（◯歳以上）や複数条件の AND/OR も表現可能。
 
## キャッシュ・即時反映の方針
 
- 「そのユーザに当たるクーポン一覧（eligibility）」のキャッシュは **BFF 側**に置く（本サービスは持たない想定）。
- **消費済み・停止中はキャッシュに焼き込まない。** 該当クーポンから表示直前に差し引くことで、TTL に関係なく即時反映する。
- 新規追加クーポンは差し引きでは出せないため、TTL 満了で自然反映（停止＝即時、追加＝多少遅れて可、の整理）。
## 関連リポジトリ
 
- `coupon-admin` — クーポン管理画面系API群（登録・編集・承認・テンプレート・CSV 等）。本リポジトリとは分離し、相互にAPIを呼ばない。
- ユーザプロファイルストア（別プロダクト・本件対象外） — 会員属性の保管庫。データチームと連携し、キャンペーン/チャンネル等でも使う横断基盤を想定。
